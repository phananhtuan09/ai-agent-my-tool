# Web QC: midnight-ai-theme-rollout

## Codebase Inventory

### Routes
- `GET /` renders the dashboard (`backend/api/pages.py`) with the `/` hero, `.dashboard-grid`, and `.dashboard-card` blocks referenced by the analyst.
- `GET /config` hosts the global OpenAI settings page plus the HTMX targets `/config/openai`, `/config/openai/test`, `/config/openai/fetch-models` defined in `backend/api/config.py` that back the form in `frontend/templates/config.html`.
- `GET /agents/{agent_name}` serves each agent console (daily scheduler, crypto airdrop) from `backend/api/pages.py`, and the base shell renders `data-stream-url="/stream/{agent_name}"`, so the SSE path is available for the connection badge.
- `/stream/{agent_name}` is referenced by `frontend/static/app.js` via the body `data-stream-url` attribute and by the EventSource client that drives `#connection-status`.

### Templates & Assets
- `frontend/templates/base.html` defines the shell (sidebar, nav links, `.topbar`, `#connection-status`, `.shell-overlay`, `data-sidebar-open`, `data-sidebar-close`) plus the `btn` classes, mobile drawer behavior, and the `Enable notifications` / `Change model` actions that must keep Midnight AI tokens.
- `frontend/templates/dashboard.html` provides `.page-hero.accent-blue`, `.dashboard-grid`, `.status-pill`, `.metric-list`, and the `.btn-ghost.card-link` CTA; the grid uses `<article class="dashboard-card accent-...">` classes directly tied to the CSS gradients.
- `frontend/templates/config.html` contains the `.page-hero`, `#openai-config-form`/`.config-form`, `label`/`input[name="base_url"]`, `input[name="api_key"]`, `#openai-model-field`, `.modal-actions` buttons (`btn-secondary`, `btn-ghost`, `btn-primary` with `data-loading-text`), and the side panel that renders either `.tag-list` or the `.empty-state.compact-empty-state` block; HTMX swaps target `#openai-model-field` and `#config-feedback`.
- `frontend/templates/daily_scheduler.html` plus partials expose `#daily-schedule-chat-panel`, `#daily-schedule-timeline`, `.schedule-chat-form`, `.schedule-settings-form`, `textarea[name="message"]`, `button.btn-primary[data-loading-text="Planning..."]`, `button.btn-primary[data-loading-text="Saving..."]`, and HTMX `hx-target="#daily-schedule-chat-panel"`/`#daily-schedule-timeline`/`#daily-schedule-controls`.
- `frontend/templates/crypto_airdrop.html` and its partials supply `.page-hero.accent-amber`, the `.filter-toggle`/`data-filter-toggle="#airdrop-sidebar"` toggle, `#airdrop-sidebar` (with `fieldset.source-fieldset`, `[data-run-airdrop]`, and `.inline-actions`), `#crypto-airdrop-chat-panel`, `#airdrop-results-panel`, `table#airdrop-table`, and HTMX `hx-target` attributes that keep `.warning-stack`, `.summary-strip`, and `.empty-state` consistent.
- `frontend/static/app.js` wires EventSource updates for the connection badge, renders HTMX replacements for the timeline (`renderScheduleTimeline`) and airdrops (`renderAirdropCards`), manages loading indicators on buttons with `data-loading-text`, and toggles `body.sidebar-open`, `aria-expanded`, and `.shell-overlay` for the responsive drawer; it also reacts to `[data-filter-toggle]`, `[data-trigger-click]`, and `[data-focus-target]`.
- `frontend/static/style.css` defines the Midnight AI tokens (gradients, `--cyan`, `--purple`, `.btn-primary`, `.btn-ghost`, `.status-pill`, `.summary-strip`, `.data-table`, `.empty-state`, `.connection-status[data-state="connected"] .conn-dot` pulse, responsive breakpoints that show `.filter-toggle` under 1024px and convert `.sidebar` into a drawer below 767px).

## Authoring Notes
- HTMX targets `#daily-schedule-chat-panel`, `#daily-schedule-timeline`, `#daily-schedule-controls`, `#crypto-airdrop-chat-panel`, `#airdrop-results-panel`, and the config model field/feedback containers require JS-enabled requests; tests must not disable htmx/JS.
- The connection badge at `#connection-status` relies on the AppShell EventSource (`/stream/{agent_name}`) to move from `data-state="connecting"` to `connected`; allow up to ~10 seconds for the SSE handshake in automated/manual probes.
- Dynamic panels (scheduler timeline+chat, airdrop results+chat) expect backend data sets; seed a few tasks and crawled airdrops locally so the tables/summary strips render rows instead of the empty states, unless specifically verifying the empty-state messaging.
- Responsive nav verification relies on the CSS breakpoints in `style.css`: narrower than 767px the sidebar becomes a drawer, between 768-1024px the filter sidebar hides by default and uses `.filter-toggle`.

## Open Questions
- Should QC enforce capturing the connection badge in the `connected` state (which depends on SSE payloads) or is the default `connecting` text acceptable for the screenshot/test artifact?
- For the dynamic HTMX refresh steps, do we need to assert specific data values (scores, cron text) or is verifying the presence of new rows/summary strips enough for theme validation?

## Handoff Summary

### Decisions
- Cover the four routes (`/`, `/config`, `/agents/daily_scheduler`, `/agents/crypto_airdrop`) plus the shared partials that control HTMX swaps, filter/controls panels, and responsive sidebar behavior.
- Authored seven test cases that exercise the hero/card grid, responsive drawer, config form, scheduler chat/timeline (static + HTMX refresh), airdrop filters/results (static + HTMX refresh).
- Selector confidence is high because every selector cited comes directly from the inspected templates or JS modules.

### Blockers
- Proving the `connected` connection badge state requires a live SSE event stream from `/stream/{agent_name}`; if the backend does not emit a payload quickly, the badge stays `connecting`.
- Scheduler/airdrop tables expect seeded data (tasks or crawled airdrops) to surface rows; absent this data the HTMX swaps render empty-state articles, limiting how many visual states can be captured automatically.

### Open Questions
- Do we explicitly need to drive the connection badge into `connected` for the QC artifact, or can the default `connecting` state be documented as acceptable?
- Should the HTMX refresh steps assert specific numeric values (task counts, cron text) or focus solely on the visual tokens (panels, pills, tables) after the swap?
